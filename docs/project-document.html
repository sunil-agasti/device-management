00 Why This Portal — Old System vs New

The previous admin tool was a basic web form with no intelligence, no automation safeguards, and a fully manual workflow. Every interaction required the admin to remember details, copy-paste IPs, and babysit the terminal. Here’s what changed and why.

Data Entry Every field was manual every time — admin had to type employee ID, email, hostname, VPN IP, username for every single request even for repeat users Once a user exists in the database, fields auto-populate from their VPN IP. SSH probes the target machine to fetch username and hostname automatically
Intelligence Zero AI capability. Pure form-based UI with no smart features. Admin had to navigate through multiple pages and buttons Auto-populate via SSH — enter VPN IP and fields fill automatically. Smart validation prevents errors

01 System Overview

The Device Management Portal is a full-stack web application that provides a modern interface for managing temporary access privileges on Apple-managed MacBooks. It replaces manual SSH terminal operations with a centralized web portal featuring auto-populated forms, automated workflows, and real-time access tracking.

The portal runs locally on the admin's machine, authenticated via Apple VPN (17.x.x.x range), and executes bash scripts over SSH to manage remote MacBooks. A keepalive service ensures the system stays awake, VPN stays connected, and the server auto-restarts.

06 Data Flows

Admin Access — Grant Flow

1
VPN IP Lookup (on blur)SSH to target Mac via SSH_ASKPASS → fetches console username + hostname → DB lookup for employee ID + email → auto-populates form
2
Pre-check: Already Admin?Runs dseditgroup -o checkmember on target. If user is already admin, returns immediately — no grant, no log, no revoke scheduled
3
Grant Admin (streamed live)SSH → echo $PASS | sudo -S dseditgroup -o edit -a $USER -t user admin → verify with checkmember → real-time log streamed to UI via NDJSON
4
JAMF Commands (streamed live)SSH → jamf manage + jamf policy + jamf recon → updates MDM inventory, enforces policies. Also runs via background script for reliability
5
Install Revoke LaunchDaemonInstalls /usr/local/bin/admin_revoke.sh (root:wheel 700) + com.tcs.admin.revoke.plist (RunAtLoad + KeepAlive). Uses epoch timestamp, not sleep — survives reboot/shutdown/VPN disconnect
6
Notify User (dialog box)macOS dialog: “Hello {user}, you have been granted temporary admin access for {N} minutes.” with OK button. Gives up after 5 minutes if unacknowledged
7
Log & Schedule WarningWrites GRANTED to admin_logs.json. Schedules 1-minute warning notification. Server-side backup revoke at duration+1 min (failsafe)

Admin Access — Revoke Flow (Bulletproof)

1
Epoch Timer CheckLaunchDaemon runs revoke script on boot. Script loops every 30 seconds comparing date +%s vs stored expiry epoch. Fires when time is reached — correct even after reboot
2
Remove Adminecho $PASS | sudo -S dseditgroup -o edit -d $USER -t user admin
3
Verify Removaldseditgroup -o checkmember -m $USER admin — confirms user is no longer admin
4
Force Retry (if still admin)Retries dseditgroup -o edit -d up to 5 times with 2-second intervals
5
Notify User (dialog box)“** User Privileges Updated ** — Hello {user}, your admin privileges have been revoked and updated to Standard User.”
6
Self-CleanupRemoves /usr/local/bin/admin_revoke.sh, unloads and removes com.tcs.admin.revoke.plist

Survives: Machine reboot, shutdown, VPN disconnect, network loss, portal server crash. Password stored securely in root-owned script (700 permissions). KeepAlive ensures launchd restarts the script if it's killed.

GitHub Access — Grant Flow

1
Pre-check: Already Accessible?Checks if github.com exists in /etc/hosts. If not blocked, returns immediately — no changes made
2
Unblock GitHubSSH → remove github.com entries from /etc/hosts → flush DNS (dscacheutil -flushcache + killall -HUP mDNSResponder)
3
JAMF CommandsBackground: jamf manage + policy + recon
4
Install Revoke LaunchDaemoncom.tcs.github.revoke.plist (RunAtLoad + KeepAlive) with epoch-based timer. Survives reboot
5
Notify User“Hello {user}, you have been granted public GitHub access for {N} minutes.”

GitHub Access — Revoke Flow

1
Re-block GitHubAdds 127.0.0.1 github.com and 127.0.0.1 www.github.com back to /etc/hosts
2
Flush DNSdscacheutil -flushcache + killall -HUP mDNSResponder
3
Notify User“** GitHub Access Revoked ** — Your public GitHub access has been revoked.”

Self-CleanupRemoves revoke script + LaunchDaemon plist

SSH Connection Method

All SSH connections use SSH_ASKPASS for password-based authentication. A temporary askpass script (/tmp/.askpass_*) echoes the password from .env. SSH is invoked with SSH_ASKPASS_REQUIRE=force which works with both password and keyboard-interactive authentication methods. The askpass file is deleted immediately after use.

Real-Time Progress Streaming

The admin access API uses NDJSON streaming (newline-delimited JSON). Each step’s status and log are streamed to the browser the moment they complete. The frontend reads the stream with ReadableStream.getReader() and updates each progress step in real-time. Users can expand any step to see the actual SSH commands and output.

Dynamic IP Handling

1
On VPN IP EntryDebounced lookup (500ms) — SSH to IP, get username (primary key) + hostname
2
DB UpdateupsertUser() updates the user's stored IP to the new one
3
Revoke Uses LatestServer-side backup revoke re-reads DB for current IP, falls back to original grant IP
4
LaunchDaemon is LocalPrimary revoke runs on the target machine itself — no SSH needed, unaffected by IP changes

Prerequisites

  • Node.js 18+ and npm
  • SSH access to managed MacBooks as tcsadmin
  • Apple VPN connection (IP starting with 17.)
  • expect (built into macOS at /usr/bin/expect)

# Clone and install
git clone https://github.pie.apple.com/s-agasti/system-admin-portal.git
cd system-admin-portal
npm install

# Initialize database
mkdir -p data data/backup
echo '[]' > data/users.json
echo '[]' > data/admin_logs.json

# Run
npm run dev          # Dev server on port 3000
npm run build && npm start  # Production

Environment Configuration

Create a .env file in the project root with the following:

# SSH Credentials (REQUIRED)
SSH_USER=tcsadmin
SSH_PRIMARY_PASS='your-primary-password'
SSH_BACKUP_PASS='your-backup-password'

# App URL
APP_URL=http://localhost:3000

# IDMS Authentication (optional)
IDMS_ENABLED=false

Note: SSH connections use SSH_ASKPASS which supports both password and keyboard-interactive authentication. The password is never exposed in logs or browser output. Use http://localhost:3000 to access (Safari does not support 0.0.0.0).

at.apple.com Setup

  1. Go to at.apple.com → Create redirect

  3. URL: http://<your-vpn-ip>:3000/device-management-portal
  4. Share https://at.apple.com/tcs-device-management-portal with your team

13 Security

Authentication & Access Control

  • VPN-Gated Access — Every request verified to originate from Apple VPN (17.x.x.x). Non-VPN traffic blocked at middleware level.
  • IDMS SSO — Optional Apple Connect single sign-on integration for identity verification.
  • Session Timeout — 15-minute idle timeout with automatic logout.
  • Rate Limiting — 200 requests/minute per IP to prevent abuse while supporting normal portal usage.

Data Protection

  • CSRF Protection — All POST/PUT/DELETE endpoints require cryptographically generated CSRF token. Cookie auto-detects HTTPS for secure flag.
  • Input Sanitization — Dedicated sanitization for IPs, hostnames, employee IDs, emails. Shell injection prevented at every boundary.
  • Credential Isolation — SSH passwords stored in .env only, never in source. Temporary askpass files created with 700 permissions, deleted immediately after use.
  • Atomic File Writes — Write-to-temp-then-rename pattern prevents data corruption. Read operations retry 3x on parse failure.
  • No Credentials in Logs — Connection logs, error messages, and browser output never contain passwords or SSH commands.

Security Headers

  • Content-Security-Policy — Restricts script and resource loading
  • Strict-Transport-Security — Forces HTTPS with max-age 1 year
  • X-Frame-Options: DENY — Prevents clickjacking
  • X-Content-Type-Options: nosniff — Prevents MIME sniffing
  • Referrer-Policy: strict-origin-when-cross-origin
  • Permissions-Policy — Blocks camera, microphone, geolocation, payment

Remote Machine Security

  • Revoke scripts — Installed as root:wheel 700 (only root can read). Contains sudo password but inaccessible to standard users.
  • LaunchDaemon — RunAtLoad + KeepAlive ensures revoke survives reboot, shutdown, VPN disconnect.
  • Self-cleanup — Revoke scripts and LaunchDaemon plists deleted after successful revocation.

Revoke Intelligence

  • Dual revoke system — Primary: LaunchDaemon on target machine (epoch-based, 10-second poll). Backup: server-side SSH revoke at duration+1 minute.
  • Smart status detection — After 3 SSH retry attempts, performs final verification check. If LaunchDaemon already revoked, marks as REVOKED (not FAILED).
  • Retry with backoff — Server-side backup retries 3 times with 30-second delay between attempts. Handles VPN reconnections and IP changes.
  • Failure logging — All revoke attempts logged to data/failure_logs.json with timestamp, IP, status, and detailed output for debugging.
  • Notifications — Fire-and-forget (non-blocking). Dialog auto-dismisses after 10 seconds. Grant, 1-minute warning, and revoke notifications.

Visitor Analytics

  • Session-based tracking — 1 visit per session. New visit only after 15-minute idle gap. Deduplication on both client (sessionStorage) and server.
  • User resolution — Local: whoami + scutil. VPN: DB lookup then SSH fallback. No anonymous visitors.
  • Reports tab — Total visits, unique visitors, daily counts, top visitors, recent visit timeline.

Hostname Management

  • Full logging — All hostname changes tracked in data/hostname_logs.json with old/new hostname, username, requestedBy.
  • Same-name validation — Prevents updating to the same hostname (client + server validation).
  • Remote notification — User receives dialog: “Your device hostname has been updated to {new}.”

Known Limitations

  • File-based DB race conditions — Mitigated with atomic writes (write-to-tmp + rename) and read retry with backoff. Acceptable for single-admin concurrent use.
  • SSH passwords in shell commands — Required for non-key authentication (keyboard-interactive). Secured via temporary SSH_ASKPASS files with 700 permissions, deleted immediately.
  • Synchronous SSH calls — Block Node.js event loop during execution (~5s per call). Acceptable for single-admin use; would need async worker pool for high concurrency.
